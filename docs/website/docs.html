Usage Guide

Practical integration patterns for production codebases. hallx works as a guardrail layer between your LLM call and any downstream action — API responses, database writes, or automation triggers.

Installation

Install hallx from PyPI. It requires Python 3.9 or later and has only two runtime dependencies.

shell
pip install hallx
shell
git clone https://github.com/dhanushk-offl/hallx cd hallx pip install -e .[dev]

Runtime dependencies:

  • jsonschema >= 4.0, < 5.0 — JSON schema validation
  • rapidfuzz >= 3.0, < 4.0 — fast fuzzy string matching for grounding

Quick Start

The simplest possible hallucination check. All fields except prompt and response are optional.

Python
from hallx import Hallx checker = Hallx(profile="balanced") result = checker.check( prompt="Summarize the refund policy", response={"summary": "Refunds are allowed within 30 days."}, context=["Refunds are allowed within 30 days of purchase."], schema={ "type": "object", "properties": {"summary": {"type": "string"}}, "required": ["summary"], "additionalProperties": False, }, ) print(result.confidence) # float 0.0 – 1.0 print(result.risk_level) # "low" | "medium" | "high" print(result.scores) # { schema, consistency, grounding } print(result.issues) # list[str] — diagnostic messages print(result.recommendation) # { action, suggested_temperature, suggestions }

Result Object

Every check() call returns a HallxResult with these fields:

FieldTypeDescription
confidencefloatWeighted aggregate score, clamped to [0.0, 1.0]
risk_levelstr"low", "medium", or "high"
scoresdictPer-signal scores: schema, consistency, grounding
issueslist[str]Human-readable diagnostic messages for each problem found
recommendationdictaction ("proceed"/"retry"), suggested_temperature, suggestions list

Scoring Model

hallx computes confidence as a weighted sum of three normalized signals:

Formula
confidence = clamp( schema_score * w_schema + # default 0.34 consistency_score * w_consistency + # default 0.33 grounding_score * w_grounding, # default 0.33 0.0, 1.0 )

Skipped checks (e.g. no context provided = no grounding check) are penalized by the profile's skip_penalty, preventing false confidence from partial analysis.

Risk Levels

Confidence rangeRisk levelRecommended action
< 0.40 high Block. Do not serve. Retry or escalate to human review.
0.40 – 0.74 medium Caution. Surface caveats, lower temperature, consider retry.
≥ 0.75 low Proceed. Safe to use in downstream automations.

Safety Profiles

Choose a profile at instantiation time. Override individual settings as needed.

Python
# Optimized for low latency — 2 consistency runs, light penalty checker = Hallx(profile="fast") # consistency_runs=2, skip_penalty=0.15
Python
# General-purpose default — 3 consistency runs, moderate penalty checker = Hallx(profile="balanced") # also the default if omitted # consistency_runs=3, skip_penalty=0.25
Python
# Maximum scrutiny — 4 runs, high skip penalty checker = Hallx(profile="strict") # consistency_runs=4, skip_penalty=0.40
Python
# Override any individual setting checker = Hallx( profile="balanced", weights={ "schema": 0.5, "consistency": 0.25, "grounding": 0.25, }, consistency_runs=5, skip_penalty=0.30, )

Sync Integration

Use checker.check() in synchronous contexts — API handlers, batch pipelines.

Python
from hallx import Hallx checker = Hallx(profile="balanced") result = checker.check( prompt="Summarize refund policy", response={"summary": "Refunds are allowed within 30 days."}, context=["Refunds are allowed within 30 days of purchase."], schema={ "type": "object", "properties": {"summary": {"type": "string"}}, "required": ["summary"], }, ) if result.risk_level == "high": raise ValueError("Response is high risk — do not serve")
Python
# Omitting context — grounding check is skipped # skip_penalty is applied to the confidence score result = checker.check( prompt="What is the capital of France?", response="Paris", # no context= or schema= provided ) # Only schema + consistency signals are evaluated print(result.scores) # { schema: 1.0, consistency: 0.95, grounding: None }
Python
# Context provided, no schema — consistency + grounding only result = checker.check( prompt="What are the side effects of ibuprofen?", response="Common side effects include nausea and stomach pain.", context=[ "Common side effects of ibuprofen include nausea, stomach pain,", "heartburn, dizziness, and mild headache.", ], # no schema= provided )

Async Integration

Use checker.check_async() in async web services and high-throughput workers. Pass an adapter for automatic response generation.

Python
import asyncio from hallx import Hallx from hallx.adapters import OpenAIAdapter adapter = OpenAIAdapter(model="gpt-4o-mini") checker = Hallx(profile="balanced", adapter=adapter) async def main(): result = await checker.check_async( prompt="What are the side effects of ibuprofen?", context=["Common side effects include nausea, stomach pain..."], ) print(result.confidence, result.risk_level) asyncio.run(main())
Python
import asyncio from hallx import Hallx from hallx.adapters import AnthropicAdapter adapter = AnthropicAdapter(model="claude-3-5-haiku-20241022") checker = Hallx(profile="strict", adapter=adapter) async def main(): result = await checker.check_async( prompt="Summarize this contract clause", context=["Contract clause text here..."], ) if result.risk_level == "high": print("Block: high hallucination risk") asyncio.run(main())

Strict Gating Pattern

Enable strict=True to raise HallxHighRiskError on high-risk responses instead of returning a result. Use on sensitive automated paths.

Python
from hallx import Hallx, HallxHighRiskError checker = Hallx(strict=True, profile="strict") try: result = checker.check(prompt="p", response="r", context=["c"]) # only reached when risk_level != "high" proceed_with(result) except HallxHighRiskError: # block output and run fallback policy return_safe_fallback()
When to use strict mode Machine-executed outputs, sensitive business workflows (financial transactions, medical advice), and customer-facing high-stakes responses.

Retry Middleware Pattern

Use result.recommendation fields to drive a retry loop — reducing temperature on each attempt.

Python
from hallx import Hallx checker = Hallx(profile="balanced") MAX_RETRIES = 3 for attempt in range(MAX_RETRIES): response = call_llm(prompt, temperature=temperature) result = checker.check(prompt=prompt, response=response, context=ctx) if result.recommendation["action"] == "proceed": break # use suggested temperature from recommendation temperature = result.recommendation.get("suggested_temperature", 0.3) else: escalate_to_human_review(prompt, result)

Feedback & Calibration

Record human-reviewed verdicts to a local SQLite database, then run calibration reports to tune thresholds to your real-world data.

Python
from hallx import Hallx checker = Hallx(feedback_db_path="/var/lib/myapp/hallx.sqlite3") result = checker.check(prompt="p", response="r", context=["c"]) # Record a reviewed outcome ("safe" or "hallucinated") checker.record_outcome( result=result, label="hallucinated", metadata={"reviewer": "qa-team"}, prompt="p", response_excerpt="r", ) # Get calibration report over the last 30 days report = checker.calibration_report(window_days=30) print(report["suggested_threshold"]) print(report["hallucination_rate"]) print(report["threshold_metrics"])

Known Limitations

Important hallx is heuristic and does not provide formal factual guarantees. For high-stakes domains, combine hallx with domain-specific validators and human review.
  • High confidence can still be wrong if context is missing, stale, or incorrect.
  • Similarity-based grounding checks can miss nuanced semantic contradictions.
  • Consistency checks depend on re-running the generation — latency scales with consistency_runs.
  • Does not evaluate factual accuracy against a knowledge base (no retrieval built in).
